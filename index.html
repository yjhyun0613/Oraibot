Overview

프로젝트 개요

제조 및 자동화 산업에서 구면 형태의 가공물이나 돔 형태의 구조물을 로봇으로 검사, 스캔하는 공정은 높은 정밀도와 안정성을 필요로 합니다.

본 프로젝트는 두산 협동로봇(DSR-M0609)을 이용하여 돔 형상의 표면을 마찰이나 파손 없이 순응 제어(Compliance Control) 기법으로 미끄러지듯 스캔하고, 획득한 실시간 3D 위치 데이터를 클라우드 및 대시보드로 전달하여 원격 품질 분석을 가능하게 하는 디지털 트윈 기반 모니터링 시스템입니다.

수집된 실시간 궤적을 사전에 설계된 이상적인 돔 형상 데이터(Baseline)와 KDTree 데이터 구조를 통해 초고속으로 대조 연산하고, 5.0mm 이상의 오차가 발생한 비율(불량률)이 10.0%를 초과할 시 불량을 즉각 감지하여 리포트를 생성합니다.

"로컬 독립적 로봇 제어를 넘어 IoT 클라우드 플랫폼으로"

로컬 PC 내부에서만 동작하던 로봇 감시 시뮬레이션을 원격 클라우드(Firebase Storage 및 Realtime Database)와 결합하고 웹에 최적화된 형태로 스트리밍하여, 원격에 있는 관리자가 현장의 품질 상태를 실시간 디지털 트윈으로 제어 및 감시할 수 있도록 확장하였습니다.

6번 관절 꼬임 및 특이점(Gimbal Lock) 에러

발생 현상 (Symptom)

구면 표면을 지속적으로 돌며 스캔할 때, 6번 손목 관절 각도가 한계치에 다다르거나 오일러 각 변환 도중 특이점(Singularity)에 빠져 로봇이 중간에 구동을 멈추고 뻗는 현상.

원인 분석 (Root Cause)

로봇이 한쪽 방향으로만 회전하면 관절 꼬임이 누적됩니다. 또한 툴의 방향 벡터를 RX, RY, RZ 오일러 각으로 단순히 매핑 시, 수학적으로 분모가 0이 되어 무한대 회전 명령이 생성됩니다.

해결 방안 (Solution)

① 지그재그 회전 방식 도입: 홀수 층은 시계 방향(CW), 짝수 층은 반시계 방향(CCW)으로 돌도록 DRL을 지그재그 패턴으로 설계하여 이전 층에서 꼬인 관절을 다음 층에서 스스로 풀도록 유도했습니다.

② Z-Y-Z 오일러 각 변환(rot_to_zyz) 및 짐벌락 방어: Direct 6축 매핑 대신 회전 행렬을 오일러각으로 변환하는 수식을 DRL 내부에 직접 이식하고, 특이값 영역을 선제적으로 예외 처리하는 코드를 삽입했습니다.

# rot_to_zyz 함수 적용 예외 필터링 예시 if abs(r22) > 0.9999: # 짐벌락 검출 시 ay = 0.0 az = math.atan2(-r01, r00) else: ay = math.atan2(math.sqrt(r02**2 + r12**2), r22) az = math.atan2(r12, -r02) # 홀/짝수 층별 CW/CCW rz 시작각 교차 분배로 리미트 방어

층간 이동 시 모션 딜레이 및 버벅거림

발생 현상 (Symptom)

스캔 중 층(Layer)을 이동할 때마다 로봇이 공중에서 완전히 정지했다가 다음 목표 좌표를 향해 가속하여 속도가 급격하게 저하되고 덜컥거리는 현상.

원인 분석 (Root Cause)

이전 스크립트에서는 로봇의 다음 명령이 전달되기 전에 wait_for_idle()로 완전 정지 대기 상태를 가졌으며, DRL 모션 큐에 연속적으로 위치 데이터가 들어가지 않아 발생한 문제였습니다.

해결 방안 (Solution)

논스톱 모션 블렌딩(Non-stop Blending): 강제 동기화 대기를 전면 삭제하고, 두산 DRL의 movel 명령어에 블렌딩 반경 옵션(r=15.0)을 추가하여 목표 좌표 지점을 통과할 때 속도를 줄이지 않고 부드러운 곡선 궤적으로 흘러가도록 제어 아키텍처를 변경했습니다.

# 층간 이동 시 블렌딩 적용 DRL 생성 로직 # movel(target_point, v=veloc, a=accel, r=15.0) # wait_for_idle() 제거 -> DRL 모션 버퍼 큐에 연속 push

로봇 헤드 충돌 및 돔 표면 파손 위험

발생 현상 (Symptom)

수학적 계산상의 3D 구면 좌표와 실제 플라스틱 돔의 물리적 위치가 미세하게 다를 경우, 로봇 헤드가 돔 표면을 과하게 찍어 눌러 파손되거나 스캐너가 들뜨는 문제.

원인 분석 (Root Cause)

이상적인 원(Sphere) 구동은 로봇 툴을 완벽히 딱딱한 물리적 벽(Hard Constraint)으로 구동하므로, 실제 조립 공차나 곡면 오차(1~2mm)가 존재할 경우 과도한 힘이 작용하게 됩니다.

해결 방안 (Solution)

순응 제어(Compliance Control) 및 가압 오프셋 적용: DRL 내부에서 힘 제어 모드(task_compliance_ctrl)를 가동하여 로봇 헤드에 가상의 스프링 강성(Stiffness)을 부여했습니다. 특히 돔 구면에 접촉하는 방향의 강성을 50으로 유연하게 주어, 로봇이 돔의 표면 오차에 맞춰 흐물거리며 쓸고 지나가도록 구현했습니다. 더불어 목표 궤적을 돔 안쪽으로 2~4mm 가압(Press) 삽입하여 일정한 접촉력이 항상 작용하게 만들었습니다.

# 순응제어 (X축: 50의 부드러운 힘, Z축: 300의 강성) stiff = [50, 3000, 300, 300, 50, 300] task_compliance_ctrl(stiff) # 가압 오프셋 적용 궤적 계산 r_pressed = ACTUAL_RADIUS - PRESS_OFFSET # 돔 안쪽으로 약간 진입

웹 대시보드 내 공중 이동 '허공 거미줄' 궤적

발생 현상 (Symptom)

웹상의 3D 디지털 트윈 화면에 돔 형태의 깔끔한 스캔 궤적만 남아야 하는데, 로봇이 층을 바꿀 때나 홈 포지션에서 진입할 때 공중을 거쳐 이동하는 불필요한 직선 데이터가 거미줄처럼 그려져 시인성이 극도로 저하되는 현상.

원인 분석 (Root Cause)

단순히 전체 구동 스위치(checking_state) 토픽만 구독하여 Firebase로 데이터를 보냈기 때문에, 로봇이 허공에 떠 있는 상태에서도 좌표 전송이 멈추지 않고 중계되었기 때문입니다.

해결 방안 (Solution)

이중 스위치(Dual-Switch) 통제 방식 적용: 로봇 스캔이 활성화되었음을 뜻하는 checking_state와 실제 돔 표면에 완전히 달라붙어 밀착 궤적 이동 중일 때만 `True`가 되는 record_enable 토픽을 새롭게 정의했습니다. Firebase 중계 노드는 두 스위치가 동시에 `True`인 순간의 좌표만 선별하여 클라우드로 쏩니다.

# Firebase 데이터 중계 제어 로직 (이중 스위치) def joint_states_callback(self, msg): if self.checking_state and self.record_enable: # 10Hz 스로틀링 연산 수행 후 Firebase push self.send_to_firebase(xyz_coords)

일시적 오프셋 감지 시 강제 비상 정지 문제

발생 현상 (Symptom)

검사 중 노이즈나 통신 진동으로 인해 1개의 포인트만 5.0mm 오차 범위를 넘어서도 로봇 시스템이 강제로 비상 정지하며 연속 작업이 불가능해지는 치명적인 정지 오버헤드 발생.

원인 분석 (Root Cause)

초기 버전(`monitoring_00.py`)에서는 실시간 오차가 임계치 이상으로 뛰는 즉시 로봇에 강제 정지 서비스(stop)를 요청하도록 코드가 설계되어 일시적 신호 튀어오름에 지나치게 방어적이었습니다.

해결 방안 (Solution)

통계적 사후 불량 판정(Post-Scan Statistical Analysis): 즉시 제어 개입을 삭제하고, 로봇이 돔의 전체 층을 돌아 스캔을 종료할 때까지 궤적 좌표들을 누적 수집하도록 변경했습니다. 스캔이 정상적으로 끝난 뒤 전체 포인트 중 오차가 5.0mm를 넘긴 '불량 포인트의 비율(%)'을 연산하고, 해당 불량률이 10.0% 임계치를 넘을 때만 최종적 `FAIL` 판정을 내리도록 모니터링 분석 알고리즘을 혁신했습니다.

# 사후 판정 알고리즘 fail_points = np.sum(errors > 5.0) fail_rate = (fail_points / total_points) * 100.0 if fail_rate > 10.0: self.inspection_result = "FAIL" else: self.inspection_result = "PASS"

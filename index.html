AMR Architecture

프로젝트 기술 개요

기존의 자율주행 모바일 로봇(AMR)은 사전에 매핑된 지도를 기준으로만 이동하기 때문에, 갑작스럽게 동적인 장애물(사람, 떨어지는 상자)을 만났을 때 즉각 대처하지 못하거나 강제로 시스템이 종료되어 주행 흐름이 완전히 끊기는 한계가 있었습니다.

오라이봇(Oraibot) 프로젝트는 터틀봇 4(TurtleBot 4) 플랫폼을 기반으로, YOLOv8 딥러닝 객체 검출과 라이다 센서 액티브 필터링을 융합하여 주행 흐름의 연속성을 유지하면서 충돌 위험을 원천 차단하는 지능형 안전 감시 솔루션을 개발했습니다.

위험물이 60cm 이내의 주행 경로 영역에 진입하면 로봇은 즉각적으로 Nav2 자율주행 명령을 보류(Cancel)하고, 몸체를 장애물 방향으로 틀어(Rotate) 위험 요소를 카메라 정면에 확보합니다. 관제 화면에 경고(Alert)를 발송하고 대기하다가, 위급 상황 해제 후 원터치 재개(`r`) 신호가 전달되면 이전 목적지를 다시 기억해내서 주행을 안전하게 이어갑니다.

"위험 인지(YOLOv8) ➡️ 자율 제동/추적(LiDAR) ➡️ 관제 및 원터치 주행 재개"

OAK-D 카메라의 RGB-D 뎁스 매핑 정보와 라이다 데이터를 병렬로 분석하여 다중 방어막을 형성하고, Nav2 액션 취소 서비스 및 `cmd_vel` 다이렉트 제어권을 획득하여 긴급 정지 및 지향성 감시 제어를 안전하게 구현했습니다.

Nav2 구동 중 cmd_vel 전송 시 씹힘 현상

발생 현상 (Symptom)

라이다 센서가 장애물을 감지하여 `cmd_vel` 토픽으로 정지(0,0,0) 속도를 계속 전송했음에도 불구하고 로봇이 정지하지 않고 계속 목표치를 향해 주행하는 충돌 위험 상황 발생.

원인 분석 (Root Cause)

Nav2의 내장 로컬 플래너(Controller Server)가 실시간으로 계속 주행 속도를 계산하여 발행하기 때문에, 사용자가 발행하는 제동용 `cmd_vel` 토픽이 뒤섞여 씹히는 현상이 원인이었습니다.

해결 방안 (Solution)

Nav2 액션 강제 취소 및 override 채널 생성: 장애물 검출 시 즉각적으로 Nav2의 목표 주행 액션 취소 서비스 클라이언트(navigate_to_pose/_action/cancel_goal)를 호출하여 자율주행 스케줄을 완전 취소시킴과 동시에 로봇에 0 속도를 다이렉트로 전송하여 제동 제어권을 즉각 획득하도록 개선했습니다.

# Nav2 주행 강제 취소 서비스 클라이언트 연결 및 비동기 호출 self.cancel_client = self.create_client(CancelGoal, '/robot5/navigate_to_pose/_action/cancel_goal') if self.cancel_client.wait_for_service(timeout_sec=1.0): req = CancelGoal.Request() self.cancel_client.call_async(req) self.teleop_pub.publish(Twist()) # 즉시 제동 강제 발행

라이다 감지 각도와 로봇 실제 회전 정렬 오차

발생 현상 (Symptom)

장애물이 있는 라이다 인덱스 각도를 추출하여 로봇을 회전시켰으나, 엉뚱한 허공 방향으로 회전하며 장애물 정면에 카메라를 맞추지 못하는 문제.

원인 분석 (Root Cause)

로봇의 기준 프레임(base_link)과 라이다의 고정 위치 프레임(laser_link) 간의 오프셋(예: 터틀봇4의 경우 센서가 정면 기준 시계방향 90도 회전되어 장착됨) 및 라디안 각도의 오버플로우 처리가 미비했기 때문입니다.

해결 방안 (Solution)

TF 각도 정규화 보정 적용: 획득한 라이다 장애물 데이터의 인덱스 라디안 값에 좌표계 변환 계수(-90도)를 보정하고, 계산된 값이 180도($\pi$) 경계를 지나면서 각도가 튀어 오르는 것을 막기 위해 atan2(sin, cos) 공식을 이식하여 360도 전 영역에서의 각도를 완벽하게 정형화했습니다.

# 라이다 tf -> 터틀봇 base_link tf 보정 (-90도) robot_target_angle = angle_lidar - math.radians(90) # 각도를 -pi ~ pi 사이로 강제 정규화하여 회전각 튐 방지 robot_target_angle = math.atan2(math.sin(robot_target_angle), math.cos(robot_target_angle)) # 목표 소요 시간 계산 self.angular_speed = 0.5 if robot_target_angle > 0 else -0.5 self.rotation_duration = abs(robot_target_angle) / abs(self.angular_speed)

YOLOv8 CPU 연산 처리 지연 및 제어 지연

발생 현상 (Symptom)

웹캠/OAK-D 영상을 받아 YOLOv8 딥러닝 추론을 돌릴 때 로봇 싱글 스레드 블로킹이 일어나 주행 정지 명령이 0.5초 이상 늦게 전달되어 안전 정지 기능에 중대한 위협이 발생함.

원인 분석 (Root Cause)

라이다 콜백 및 이미지 콜백이 파이썬의 싱글 글로벌 락(GIL) 스레드에 묶여 순차 처리되면서, 약 200~300ms가 소요되는 YOLOv8 CPU 추론 작업이 제어 명령 실행을 가로막았습니다.

해결 방안 (Solution)

CUDA GPU 가속화 및 ROS 2 멀티스레드 큐 아키텍처 구현: YOLOv8 모델 추론 시 CUDA GPU 연산(.to('cuda'))을 활성화시켜 지연 속도를 10ms 단위로 단축시켰습니다. 추가적으로, 이미지 획득과 제어 명령 통신 루프를 별도의 독점 스레드 콜백 그룹(MutuallyExclusiveCallbackGroup)과 Queue(maxsize=1) 버퍼로 분리하여 메인 제어 루프의 프레임 저하를 원천 차단했습니다.

# CUDA 디바이스 이식 및 비동기 큐 처리 self.model = YOLO('yolov8n.pt').to('cuda') self.image_queue = Queue(maxsize=1) # 스레드 분리 self.thread = threading.Thread(target=self.detection_loop, daemon=True)

로봇 몸체(Chassis) 부품 인식 오작동

발생 현상 (Symptom)

로봇이 정면에 장애물이 전혀 없고 주변이 텅 빈 실내 구역임에도 불구하고 수시로 "장애물 발생"으로 급제동하며 주행을 거부하는 현상.

원인 분석 (Root Cause)

라이다 센서의 360도 레이저 빔 중 일부가 로봇 본체(섀시)나 상단에 설치된 액세서리에 부딪혀 반사되면서, 5~10cm 수준의 매우 가까운 고스트 장애물로 감지되었던 것이 원인이었습니다.

해결 방안 (Solution)

최소 유효 수집 거리(Cut-off) 필터 설정: 로봇 중심 원점으로부터 자신의 몸체 반경인 20cm 미만의 센싱 데이터를 무시하고 제거하는 안전 컷오프(self.min_distance = 0.20) 로직을 이식하여 로봇 내부에서 자체 발생한 고스트 반사 신호를 완벽하게 필터링했습니다.

# Lidar 수집 범위 중 20cm 이하 극단거리는 로봇 몸체로 판정하여 제외 self.min_distance = 0.20 # 20cm (차체 노이즈 차단) self.max_distance = 0.60 # 60cm (비상 정지 유효 영역) for i, dist in enumerate(msg.ranges): if self.min_distance < dist < self.max_distance: # 이 범위에 들 때만 진짜 장애물로 통계 처리

Nav2 취소 시 목적지 데이터 증발 문제

발생 현상 (Symptom)

장애물 발생 시 비상 브레이크 작동 및 Nav2 명령 취소는 성공했으나, 장애물이 치워진 뒤 로봇이 원래 어디로 향하고 있었는지 그 목적지를 잃어버려 자율 재주행이 불가능했던 현상.

원인 분석 (Root Cause)

Nav2의 내장 스택은 CancelGoal 신호를 받는 즉시 현재의 Plan을 메모리에서 해제하여 메모리 초기화를 시키므로 로봇 입장에서 복귀 타겟이 사라지게 되는 구조였습니다.

해결 방안 (Solution)

이벤트 기반 목적지 캐싱 및 재배포 장치 구현: 사용자가 Rviz나 관제 프로그램에서 목표 좌표(PoseStamped)를 지정하는 `/robot5/goal_pose` 토픽 채널을 백그라운드에서 상시 감시하여 별도의 로컬 캐시 변수(self.saved_goal)에 백업해 두도록 로직을 추가했습니다. 주행 재개(`RESUME`) 신호가 전달되면 이 캐시 변수로부터 목적지 좌표를 그대로 가져와 다시 릴레이 재발행함으로써 자율 주행의 연속성을 되살려 냈습니다.

# 목표 데이터 가로채기형 캐시 저장 def goal_callback(self, msg): self.saved_goal = msg # 복구용 캐시 # GUI 복구 명령 시 def resume_callback(self, msg): if msg.data == "RESUME" and self.saved_goal: self.goal_pub.publish(self.saved_goal) # 목적지 자동 재전송!
